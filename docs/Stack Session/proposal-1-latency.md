## B. Coherence check

One language (TypeScript) runs the whole system: Fastify serves the SSE stream and schema-validated command routes; the custom Scene-Engine, ledger workers, croner CRON, grammY/wechaty connectors, `web-push`, sharp, and better-sqlite3 all live in the same Node process, so every lease, idempotency key, mailbox write, and event append shares one WAL transaction boundary — which is what makes crash-only recovery and the ledger design actually atomic. The Svelte frontend is compiled to static files served by the same Fastify instance and consumes only the MIT-licensed `@weltari/protocol` JSON Schemas — the same contract the V1.5 CLI and future external game clients use, so "frontend as just another client" is enforced by construction. Plugins ship one zip containing an ESM backend half and a precompiled ESM frontend half in the same language and toolchain; the default PixiJS map renderer eats its own dogfood as such a plugin. Packaging is one `node_modules` tree (two native addons: better-sqlite3, sharp — both prebuilt for all four targets) inside either a Docker image or a runtime-bundled archive. **Estimated idle RAM:** Node baseline ~45 MB + Fastify/app code ~30 MB + better-sqlite3 caches ~20 MB + connectors idle ~15 MB + libvips lazy ~10 MB ≈ **120–150 MB idle**, comfortably under the 256 MB typical target with headroom left for plugins.

## C. Constraint audit (Brief §2, one line each)

1. **Event log as truth:** SSE replay reads straight from the event log via `Last-Event-ID`; all projections rebuilt from it — no second truth store exists in this stack.
2. **Job Ledger:** hand-rolled jobs table with states/leases/idempotency/dead-letter/per-world gate lives in the same SQLite as domain data — one transaction covers job commit + projection write.
3. **Single writer / mailboxes:** better-sqlite3's synchronous single write connection + per-character async mailbox queues + ESLint import fence; optimistic `version` columns in repos elsewhere.
4. **Crash-only:** claim-by-lease polling *is* the recovery path; painter uses write-fsync-rename; turn envelopes are event rows before LLM work starts. Honest strain: Node has no post-`kill -9` hook by definition — everything rests on WAL + lease expiry, which is exactly the design's intent.
5. **Server-pushed stream, render-only frontend:** SSE is push-only by nature; the command path is separate POSTs; Svelte holds zero adjudication.
6. **Prompt-prefix ordering:** custom `ContextAssembler` with byte-stable stable-first serialization is the *reason* orchestration is custom (item 3); routing (item 9) pins a character to one provider/model so its cache prefix survives across turns.
7. **Repository layer:** hand-written repos are the only SQL site; driver import restricted by lint; Postgres swap = reimplement repo interfaces.
8. **`actor_id`:** carried in every TypeBox event/command schema; schema validation rejects its absence at the Fastify boundary.
9. **Proposal pipeline:** one `Proposal` schema in `@weltari/protocol`, one Svelte diff component, approve/reject as POST commands.
10. **Corollaries:** every tool call passes a TypeBox validator before any repo write (LLM output never directly durable); knowledge-tier filter and private channel are ContextAssembler stages; world clock is an engine-owned table driving croner `run_at` computation. Honest strain: better-sqlite3's synchronous calls block the event loop — fine for ms-scale writes, but heavy FTS5/compaction queries must be chunked or pushed to a worker thread to protect streaming latency (see risk 3).

## D. Top 3 risks

1. **Prompt-cache economics miss the token budget.** What breaks: OpenRouter routes a character's calls across providers/backends, so the carefully ordered stable prefix never hits cache; daily tokens blow past the 1–5 M assumption (§7d). Likelihood: medium-high — cross-provider cache behavior is genuinely inconsistent. Blast radius: cost and the 5–10 s first-sentence budget (cold prefixes = slow prefill on 256K contexts). Mitigation: pin provider per character in the ModelRegistry (OpenRouter `provider.order` + `require_parameters`), send explicit cache breakpoints where supported, log `cached_tokens` from usage on every call, and gate V1 on the prototype's ≥80% criterion (§E).
2. **WeChat bridge breaks or bans accounts.** What breaks: `wechaty-puppet-wechat4u` rides the deprecated web protocol; its own docs warn of bans with no workaround, and post-2017 accounts often can't log in at all. Likelihood: high, eventually. Blast radius: contained by design — WeChat connector is an isolated plugin; Telegram and core are untouched (§7c accepts this trade). Mitigation: connector-as-plugin with health states surfaced in Config, documented swap path to alternative puppets (e.g. padlocal-class) without a core release, and onboarding copy that sets expectations.
3. **Event-loop stalls degrade time-to-first-sentence.** What breaks: a synchronous better-sqlite3 FTS5 query, a reflection fan-out burst, or sharp compositing on the main thread delays SSE flushes mid-narration. Likelihood: medium. Blast radius: perceived latency only — no correctness loss. Mitigation: sharp already uses libvips' own thread pool; cap ledger worker concurrency (default 2 LLM-class + 1 painter); move FTS5 search and compaction reads to a `worker_threads` read-only connection; add a stream-latency metric to dev mode from day one.

## E. De-risk prototype

**Build the naked hot path first:** Fastify + SSE stream + event log table + `ContextAssembler` + AI SDK/OpenRouter, running one scripted scene turn of 3 sequential calls (Narrator → character → narration) streamed sentence-by-sentence to a bare HTML page, with the turn envelope written through the ledger tables. No frontend framework, no map, no gateway. **Success criteria (measured, not vibed):** (a) first sentence rendered < 10 s on a 256K-class model with a ~50K-token stable prefix; (b) provider-reported `cached_tokens` ≥ 80% of the stable prefix on turns 2+ across 10 consecutive turns; (c) `kill -9` mid-call → restart → replay via `Last-Event-ID` resumes the client with zero duplicate or lost events. **Decision triggers:** if (b) fails with pinned providers, revisit item 9 toward direct per-provider SDKs with native cache control (and possibly narrow the "any endpoint" promise); if (a) fails on prefill alone, revisit the §4 context-size strategy (pre-retrieval aggressiveness) before touching any stack choice; if (c) fails, the flaw is in the ledger claim/lease SQL — fix it before any feature work, since items 2, 7, 8 all stand on it.
