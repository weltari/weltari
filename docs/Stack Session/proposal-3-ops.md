## B. Coherence check

Everything is one TypeScript codebase on one embedded runtime: Bun compiles the engine (Hono + custom Scene-Engine + hand-rolled ledger on `bun:sqlite`) into a single per-platform binary that also serves the Vite-built React frontend as static files and the SSE stream; plugins are directories of TS (backend, run natively by Bun) + ESM/CSS (frontend, loaded via import map) — the same two artifacts the core itself is made of, so the plugin story, packaging story and language choice are the same decision seen three times. The MIT `protocol/` JSON-Schema folder is what the React client, the V1.5 CLI and future external games all consume, keeping the built-in frontend "just another client" (§1). **Idle RAM estimate:** Bun runtime + Hono ≈ 45–70 MB, SQLite page cache ≈ 10–20 MB, grammY long-poll + croner + idle worker pool ≈ 10 MB, sharp lazy-loaded (0 idle) → **core idle ≈ 80–110 MB**, peaking ~180–220 MB during reflection fan-out + one painter composite — inside the 256 MB typical target with plugin headroom (§1).

## C. Constraint audit (Brief §2, one line each)

1. **Event log as truth:** `event_log.append` is the only durable entry point; SSE replay and all projections rebuild from it — SSE `Last-Event-ID` replay depends on it, so the stack *enforces* it.
2. **Job Ledger:** item 8 implements states/idempotency-keys/leases/retries/dead-letter/per-world concurrency literally as columns and one claim query.
3. **Single writer per entity:** one write connection behind a mutex facade; per-character mailboxes are ledger jobs keyed `mailbox:<char_id>` claimed one-at-a-time; `version` columns checked in repository update methods.
4. **Crash-only:** synchronous `bun:sqlite` transactions make intent durable before work; migrations, update-swap, plugin scan and lease-reclaim all live in the startup path; no shutdown hooks are correctness-bearing.
5. **Server-pushed stream, render-only frontend:** SSE + POST; React holds no game state beyond render caches; honest strain: SSE is one-way, so *typing* input signals also go over POST — acceptable since input is command-shaped.
6. **Prompt-prefix ordering:** owned by our `ContextAssembler`; the AI SDK receives pre-ordered message arrays and never reorders.
7. **Repository layer:** raw connection is module-private; only repositories compile — Postgres swap = new facade + repo drivers.
8. **`actor_id`:** a required field in every event JSON Schema in `protocol/`; schema validation at append time rejects its absence.
9. **Proposal pipeline:** one `Proposal` JSON Schema, one React `<ProposalDiff>` component, approve/reject as POST commands, both logged.
10. **Corollaries:** engine validates every tool call before append (custom loop, item 3); knowledge-tier filtering lives in `ContextAssembler`; world clock is an engine-owned SQLite row driving `run_at` on game-clock CRON jobs.

Honest strain: `bun:sqlite`'s synchronous writes briefly block the event loop — mitigated by keeping transactions tiny and WAL/NORMAL; if a profiler shows stream stutter during fan-out, the write facade moves to a worker thread without touching repository callers.

## D. Top 3 risks

1. **Bun maturity edge cases** (native-addon embedding for sharp, long-lived SSE stability under memory pressure on NAS boxes). *Likelihood:* medium. *Blast radius:* item 1 — the runtime itself. *Mitigation:* stick to pure-JS deps + `bun:sqlite`; ship sharp's `.node` beside the binary (not embedded); the code is standard TS + node-API, so the escape hatch is "same code on Node 22 + better-sqlite3 + pkg", a packaging change, not a rewrite — validate in the E prototype.
2. **WeChat personal-account bridge breakage/ban** (puppets are unofficial and volatile). *Likelihood:* high, eventually. *Blast radius:* one connector only — by design (§7c). *Mitigation:* `GatewayConnector` isolation + health status in Config + dead-letter lane for undeliverable pushes; core and Telegram unaffected; connector ships as a replaceable plugin from day one.
3. **Import-map plugin frontend proves too weak for deep UI plugins** (a map replacement needing app-level context beyond documented connectors). *Likelihood:* medium. *Blast radius:* items 5/6/13 plugin story. *Mitigation:* build the *default* map renderer strictly against the public connector schema (dogfooding, UI Spec §1.8); if it can't be built that way, the connector schema — not the plugin mechanism — is what gets extended, before any third party depends on it.

## E. De-risk prototype (build first, ≤1 week of AI-agent work)

**A "kill-torture" vertical slice:** Bun-compiled single binary (Windows + Linux) running Hono + SSE + `bun:sqlite` ledger, executing a scripted 3-call scene turn through the AI SDK against OpenRouter with the stable-prefix `ContextAssembler`, one sharp crop→feather→composite painter job under a lease, and a grammY long-poll echo — while a harness `kill -9`s the process at 10 random points (mid-turn, mid-composite, mid-claim) in a loop, restarting it each time.

**Success criteria (measurable):** 100 kill/restart cycles with zero lost or duplicated durable events and zero corrupted images (hash-verified); reconnecting SSE client replays to identical state via `Last-Event-ID`; first streamed sentence < 10 s; OpenRouter prompt-cache hit ≥ 80% on calls 2–3; idle RSS < 120 MB on both platforms.

**Decision triggers:** RSS > 180 MB idle or sharp/native embedding fails on Bun → revisit item 1 (fall back to Node 22 + better-sqlite3 packaging, same code); any replay divergence → revisit item 4's sequence-id design before the protocol folder is published; cache hits < 80% → revisit item 9's assembler, not the SDK.
