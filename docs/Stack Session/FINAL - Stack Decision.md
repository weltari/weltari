# Weltari V1 — Final Stack Synthesis

*Synthesized from 4 independent proposals + their fact-check verdicts, against the Stack Requirements Brief and UI Spec (skeleton) only.*

**A note on the fact-checks before anything else.** The four fact-check reports contradict each other on one legal point, and it matters. Proposal 1's checker marked every Apache-2.0 package (sharp, the Vercel AI SDK, wechaty) and the MPL-2.0 `web-push` package as "license-conflict with AGPLv3." That verdict is wrong: the Free Software Foundation explicitly lists Apache 2.0 as one-way compatible with (A)GPLv3 — Apache-licensed libraries may be used inside an AGPL program — and the checkers for Proposals 3 and 4 correctly say so. MPL-2.0 likewise contains a built-in "secondary license" clause naming AGPLv3 as compatible. I verified `web-push` really is MPL-2.0 (the checkers disagreed MIT vs MPL): the [package.json in the official repo](https://github.com/web-push-libs/web-push/blob/master/package.json) confirms MPL-2.0. So **no proposal is disqualified on license grounds**; the real fact-check casualties are *maintenance*: `@grammyjs/runner`, `leaflet.markercluster`, `wechaty-puppet-wechat4u`, `wechaty-puppet-padlocal`, and (staleness only) `web-push`. Those are handled below. One caveat: Apache-2.0 code cannot be *copied into* the MIT-licensed edge packages (`@weltari/protocol`, plugin SDK) — depending on it is fine, embedding its source is not. That's a repo-hygiene rule, not a blocker.

No proposal violated a decided item: none proposed Postgres, dropped the repository layer, or reopened the license. Proposal 4's Drizzle ORM adds a layer *on top of* the mandated repository layer (a design disagreement, handled in §2, not a violation).

## 1. Convergence map (≥3 of 4 agree — treated as settled)

| Item | Settled choice | Shared rationale (one sentence) |
|---|---|---|
| Language | **TypeScript 5.x, strict mode** (4/4) | One language across engine, frontend, and both halves of every plugin serves the brief's top two priorities (LLM ecosystem, plugin-authoring ease) at once. |
| Runtime family | **Node.js LTS** (3/4; Proposal 3 chose Bun) | Mature long-term-support releases and battle-tested native modules fit an always-on appliance better than a younger runtime. |
| HTTP framework | **Fastify 5** (3/4) | Built-in JSON-Schema validation on every route means malformed commands are rejected at the door — which is exactly the "engine validates everything" constraint (Brief §2.10). |
| Orchestration | **Custom Scene-Engine loop, no framework** (4/4) | Agent frameworks own the loop's state and prompt order, which would fight the event-log-as-truth and prompt-prefix-ordering constraints (Brief §2.1, §2.6). |
| Event transport | **SSE (server-sent events — a one-way push over plain HTTP) for events + POST for commands**, reconnect via `Last-Event-ID` replaying from the event log; schema = TypeBox types compiled to JSON Schema in an MIT `@weltari/protocol` package (4/4) | The stream is one-way by design (frontend renders only), SSE survives NAT/proxies, and `curl` can consume it — satisfying the terminal-client and engine-as-a-service mandates (Brief §1). |
| SQLite driver + discipline | **better-sqlite3, WAL mode, one write connection, repositories are the only module allowed to touch it (enforced by a lint rule)** (3/4; Proposal 3's `bun:sqlite` was the same pattern on Bun) | A synchronous driver on one event loop makes single-writer discipline structural rather than a convention (Brief §2.3, §2.7). |
| Job Ledger | **Hand-rolled jobs table in the same SQLite file + `croner` for cron parsing** (4/4) | Leases, idempotency keys, and dead-letter states must commit in the *same transaction* as domain writes — any external queue (Redis, Postgres-based) breaks that atomicity and the single-process rule (Brief §2.2, §1). |
| LLM layer | **Vercel AI SDK (`ai`) + `@openrouter/ai-sdk-provider` + `@ai-sdk/openai-compatible`, with prompt assembly 100% owned by our own `ContextAssembler`** (4/4) | The SDK normalizes every provider's streaming and tool-call quirks while never touching prompt order, which stays ours for cache-hit discipline (Brief §2.6, §3). |
| Image pipeline | **sharp (libvips)** (4/4) | Crop → composite → feather at ~1024² with prebuilt binaries on all four platforms, at RAM levels that fit the 256 MB envelope (Brief §3). |
| Telegram | **grammY, long-polling** (4/4) | Actively maintained, TypeScript-native, and outbound-only polling matches the NAT-first requirement (Brief §7c). |
| WeChat approach | **wechaty behind a swappable `GatewayConnector` interface, shipped as a plugin** (4/4) | The brief accepts that personal-account bridges are fragile, so the design isolates breakage to one replaceable plugin (Brief §7c). |
| Packaging + update | **Docker multi-arch image (primary) + per-OS archives bundling a pinned Node runtime and a launcher; update = startup/CRON check of GitHub Releases → in-UI notice; native apply = download, verify hash, swap a `current` pointer on restart; Docker = notify-and-let-host-pull** (4/4) | Crash-only design makes "swap on restart" free, and no resident updater spends RAM (Brief §1, Q6). |
| Plugin format | **A plugin is a folder (distributed as .zip): `plugin.json` manifest with semver, engine-version range, capabilities, and provenance (source URL + SHA-256 hash); backend = ES module exporting `register(api)`; frontend = ES modules + `theme.css` token overrides; `structure.md`/`skill.md` for AI-agent editing** (4/4) | One language means one zip carries both halves with no build toolchain for hobbyist authors, and the hash satisfies the provenance constraint (Brief §2 P8, §5). |
| Browser notifications | **Web Push (VAPID — a standard where your server pushes to the browser vendor's relay, no third-party service of ours) on HTTPS origins; graceful fallback to in-app toasts on plain-HTTP LAN; Telegram/WeChat remain the true away-from-keyboard channel** (4/4) | Web Push is outbound-only from the server (NAT-perfect), and the gateway already covers the closed-laptop case (Brief §3, §7c). |
| Frontend build tool | **Vite** (4/4, versions corrected by fact-check to **Vite 8**) | Standard, fast, actively maintained static build served by the same Fastify process. |

## 2. Divergence analysis

**Runtime: Node (P1, P2, P4) vs Bun (P3).** Proposal 3's ops/distribution lens picked Bun for the single-binary story. But the brief ranks distribution *last* (§7b), and Bun's win only exists under that lens — this is textbook lens bias. Against hard constraints, Bun is weaker: native-addon embedding for sharp is flagged as P3's own #1 risk, and an always-online appliance needs the most boring runtime available. P3 even names "fall back to Node" as its escape hatch. **Node wins.** On version: P1/P2 say Node 24 (current even LTS), P4 says 22 (one LTS behind, extra conservatism). No cited constraint supports staying behind; Node 24 is confirmed LTS with maintenance into 2028. **Node 24 LTS.**

**HTTP framework: Fastify (3) vs Hono (P3).** Hono's advantages (tiny, Bun-native) are riders on the Bun choice, which lost. Fastify's schema-validation-per-route directly services Brief §2.10. **Fastify wins with the majority.**

**Frontend framework: Svelte (P1) vs Lit (P2) vs React (P3, P4).** The genuinely contested item. Each lens shows: P1 (latency) liked Svelte's small compiled output; P2 (plugin ecosystem) liked Lit because plugin components need no build step; P3/P4 liked React because the UI Spec (§1.13) explicitly wants a frontend *editable by third-party AI agents*, and React has by far the deepest AI-editing track record. Scrutiny against the brief's ranking (ecosystem > plugins > familiarity > distribution): Svelte loses because *both* other camps independently identified the same hard problem — Svelte plugins must be compiled with a matching compiler version, which couples every plugin to the app's build, hurting the #2 priority. P2's Lit argument is real but is fully satisfied by a hybrid P4 already designed: **React core, with the plugin boundary being framework-neutral native custom elements** (`<wl-map>`, `<wl-*>` tags — a browser standard, no framework needed). Plugin authors write plain zero-build JavaScript (P2's win) while the core gets React's AI-editability and ecosystem (P3/P4's win). Fact-check correction absorbed: Vite 6/7 are obsolete → **Vite 8**. **Final: React 19 + Vite 8, custom-element plugin boundary, CSS custom-property theming, CSS/Web-Animations for VN motion.**

**Map renderer: PixiJS (P1) vs Leaflet (P2) vs hand-rolled Canvas 2D + DOM overlay (P3, P4).** Leaflet is eliminated on facts: its clustering companion `leaflet.markercluster` is fact-checked as unmaintained (5 years), Leaflet core is in maintenance mode, and its geographic-coordinate assumptions fight world-coordinate pins — the "most familiar library" argument is a plugin-lens preference, not a constraint. PixiJS (confirmed maintained) vs Canvas 2D is closer: P1's WebGL-performance argument is real *at extreme tile counts*, but it's speculative where Canvas 2D's advantages are concrete — zero GPU context-loss handling, lower RAM against the 256 MB target, and a simpler reference implementation for the one component the brief demands be *wholesale replaceable by third parties* (UI Spec §1.8). Because the map renderer is a plugin by decree, a PixiJS renderer can be the first community plugin if Canvas 2D ever janks. **Canvas 2D tiles + DOM overlay for pins/spinners, packaged as the default `<wl-map>` plugin (2-vote plurality + constraint fit).**

**SQL access: hand-written repositories (P1, P2, P3) vs Drizzle ORM under the repositories (P4).** P4's solo-maintainer lens values Drizzle's compile-time-checked queries. But the brief already mandates a repository layer as *the* seam (§2.7); an ORM (object-relational mapper — a library that writes SQL for you) underneath it is a second query dialect that must also survive the future Postgres swap, exactly the objection three proposals raised independently. Typed safety is recoverable more cheaply: TypeScript types on repository interfaces + tests. **Hand-written repositories over prepared statements win 3–1.**

**Telegram concurrency: `@grammyjs/runner` (P1) vs grammY's built-in long-polling (P2, P3, P4).** The runner package is fact-checked unmaintained (2 years, single maintainer). A single-user app does not need a high-concurrency update runner anyway. **Drop the runner; use grammY's built-in `bot.start()` polling.**

**WeChat puppet: `wechaty-puppet-wechat4u` (P1, P2, P4) vs padlocal-class (P3).** *Both* are fact-checked unmaintained — there is no verified, maintained, free personal-account WeChat bridge; that is the reality the brief already accepted (§7c: "unofficial and ToS-fragile"). Under the rejection rule neither puppet can be a load-bearing core dependency, but the brief's decided item (WeChat ships V1 via such a bridge) is binding. Resolution: wechaty *core* is maintained and license-fine; the puppet ships as the default inside an **isolated, clearly-labeled experimental connector plugin** with wechat4u (free, QR-pairing, matching the brief's description) as the default and padlocal documented as the paid swap. This is also surfaced as Owner Decision #1.

**web-push package.** Verdicts conflicted; verified: MPL-2.0 (AGPL-compatible via its secondary-license clause) but ~2 years stale. The Web Push protocol itself is a frozen IETF standard, so staleness is low-risk; mitigation is to pin the version and keep the Apache-2.0 `node-webpush` alternative (named by fact-check 1) documented as the swap. **Keep `web-push`.**

## 3. Final stack

| # | Item | Final choice |
|---|---|---|
| 1 | Backend language + runtime | **TypeScript 5.x (strict) on Node.js 24 LTS**; even-numbered LTS only, pinned via `.node-version` + `engines`, dependencies pinned exact with monthly batched updates |
| 2 | HTTP framework | **Fastify 5** + TypeBox JSON-Schema validation on every command route; serves the static frontend and the SSE stream from the same process |
| 3 | Orchestration | **Custom Scene-Engine loop** — plain state machine (turn envelope → sequential LLM calls → validated tool commits → close) + router, owned code |
| 4 | Event transport + schema | **SSE (`GET /v1/events`) + POST commands**; event `id:` = event-log sequence; reconnect = browser-native `Last-Event-ID` replay from the log; dev-mode events as a gated channel on the same stream; schema = TypeBox → JSON Schema in MIT **`@weltari/protocol`**, documented as OpenAPI 3.1 (commands) + AsyncAPI 3 (events), semver'd with `protocol_version` in the handshake |
| 5 | Frontend | **React 19 + Vite 8**; zustand stores writable only by the SSE reducer (enforces render-only); **plugin boundary = native custom elements (`<wl-*>`)** so plugin components need zero build step; per-world theming = CSS custom-property token files; VN animation = CSS transitions + Web Animations API; dedicated mobile layouts on shared stores; `structure.md`/`skill.md` in-repo |
| 6 | Map rendering | **Hand-rolled Canvas 2D tile/fog/lasso layers + DOM overlay for pins, clusters, and per-square spinners**, shipped as the default `<wl-map>` plugin consuming only documented map connector events (dogfoods the plugin contract) |
| 7 | SQLite access | **better-sqlite3 v12**; `WAL`, `synchronous=NORMAL`, `busy_timeout=5000`, `foreign_keys=ON`; one write connection behind a WriteGate; **hand-written repositories** are the only SQL site (ESLint `no-restricted-imports` fence); per-character mailboxes = in-process serial queues; numbered `.sql` migrations via `PRAGMA user_version` applied at startup |
| 8 | Job Ledger runner | **Hand-rolled on SQLite**: `jobs` table with idempotency-key UNIQUE, states, leases, attempts→`parked` dead-letter, per-world concurrency in the claim query; ~4 async workers; expired-lease sweep on the same poll (startup *is* recovery); **croner** computes cron `run_at`, always writing ledger rows, never working inline |
| 9 | LLM layer | **Vercel AI SDK (`ai` v5; evaluate v6 at prototype time) + `@openrouter/ai-sdk-provider` + `@ai-sdk/openai-compatible`**; `ModelRegistry` maps character/function → provider+model+params with per-character provider pinning for cache stability; **our own `ContextAssembler`** emits byte-stable stable-first prefixes (unit-tested) + dynamic tail; sentence-split streaming to SSE; VLM via the same SDK's image parts |
| 10 | Image pipeline | **sharp**: extract → backend call → composite with blurred-alpha feather mask → resize; painter = ledger jobs with region locks, temp-file + atomic rename + event append (composite-on-success, kill-safe) |
| 11 | Gateway connectors | **grammY** built-in long-polling (no runner package); **wechaty + wechat4u puppet as a bundled *experimental* connector plugin** (padlocal = documented paid swap); hand-rolled `GatewayConnector` interface (`start/stop/send/onInbound/health`); dedup via `(connector_id, external_msg_id)` UNIQUE; same `conversation_id` as in-app chat; connectors are plugins |
| 12 | Packaging + self-update | **Docker multi-arch on ghcr (primary)** + per-OS zip/tar.gz bundling pinned Node runtime, app bundle, prebuilt natives (better-sqlite3, sharp), launcher script; update check = startup + croner job → GitHub Releases → `update.available` event; native apply = download to `versions/vNext`, verify SHA-256 (+ minisign signature), flip `current` pointer on restart; Docker = notify + documented optional Watchtower; zero resident updater |
| 13 | Plugin format | **Folder/zip with `plugin.json`** (name, semver, engine range, capabilities `{skills, themes, components, connectors}`, provenance `{source_url, sha256}`), `backend/index.mjs` exporting `register(api)`, `frontend/*.mjs` custom-element ESM, `theme.css`, `skills/*.md`, `structure.md`/`skill.md`; hash verified at install and load, provenance shown in Config and dev mode |
| 14 | Browser notifications | **Web Push (VAPID) via `web-push`** (MPL-2.0 — AGPL-compatible; version-pinned, `node-webpush` documented as swap) on secure origins (localhost, Tailscale Serve, Cloudflare Tunnel); insecure LAN → SSE-driven in-app toasts + title badge, degradation surfaced in Config; Telegram/WeChat remain the true away channel |

**Combined idle-RAM estimate:** Node 24 baseline ~55–70 MB + Fastify/engine/app code ~30–40 MB + better-sqlite3 page caches ~15–20 MB + grammY long-poll + croner + idle workers ~10 MB + wechaty bridge (when paired) ~20–30 MB + sharp lazy-loaded (~0 idle, ~40–60 MB transient during painter jobs) ≈ **130–170 MB idle**, painter spikes transient, leaving plugin headroom inside the 256 MB typical target.

**Coherence check.** This is one TypeScript codebase in one Node 24 process: Fastify serves the Vite-built React frontend as static files, the SSE stream, and the schema-validated command API; the custom Scene-Engine, ledger workers, croner, grammY poller, wechaty bridge, and web-push sender are async loops in that same process, all writing through hand-written repositories into one WAL SQLite file — so every lease, idempotency key, mailbox write, and event append shares one transaction boundary, which is what makes crash-only recovery genuinely atomic rather than aspirational. The plugin story, the frontend story, and the language choice are the same decision seen three times: plugins are folders of ES modules in the one language, their frontend halves are the same zero-build custom elements the core's own map renderer is made of, and the MIT `@weltari/protocol` schemas the React client consumes are identical to what the V1.5 CLI and future external game clients will consume — "frontend is just another client" holds by construction, with no private side-channels. Packaging bundles exactly this: one JS bundle, two prebuilt native addons, one pinned runtime. Where I overruled a majority or a proposal, it was against a hard constraint or a fact-check, never taste: Bun fell to native-addon maturity and the brief's own priority ranking; Leaflet fell to an unmaintained dependency; the frontend pick is P4's hybrid because it is the only option that satisfies both the AI-editability demand (UI Spec §1.13) and the zero-toolchain plugin demand (Brief §5) simultaneously.

## 4. Owner decisions (3 — everything else is settled by constraints)

**1. How loudly do we label WeChat as fragile?** Both available WeChat bridge libraries are unmaintained (2–4 years stale) and carry a real, documented risk of the paired WeChat account being banned. The design already isolates this in a swappable plugin, so the core is never at risk — the question is purely presentation: ship WeChat as a normal V1 feature, or as a bundled connector explicitly labeled "experimental — account-ban risk, best effort"? **Recommendation: the experimental label.** It keeps your V1 promise while protecting users' expectations and their WeChat accounts; if a maintained bridge appears later, the label comes off without a release.

**2. Is a small setup step acceptable for closed-tab browser notifications?** Browsers only allow real push notifications over HTTPS, and a home NAS on plain `http://192.168.x.x` doesn't qualify — so out of the box, browser notifications work only while a tab is open, and the docs would tell users "run one command (Tailscale Serve) to unlock closed-tab push." The alternative is building nothing extra and pointing everyone at Telegram for away-notifications. **Recommendation: accept the documented setup step.** It costs you a documentation page, not code, and Telegram already covers everyone who skips it.

**3. Should characters be pinned to one provider by default?** Pinning each character's calls to a single provider makes prompt caching reliable, which is the difference between hitting your token budget (1–5M/day) and multiplying it — but it means switching a character's model mid-story resets their cache and briefly costs more. **Recommendation: pin by default, allow switching freely with a small "this will re-warm the cache" notice.** You keep the savings without taking away the freedom.

## 5. Consolidated risk register (top 5 for the final stack)

| # | Risk | Likelihood / blast radius | Mitigation |
|---|---|---|---|
| 1 | **Prompt-cache misses blow the token and latency budgets** — if OpenRouter routes a character across providers, or any AI-written edit leaks dynamic content into the stable prefix, costs multiply silently and cold 256K-context prefills eat the 5–10 s budget | Medium-high / cost + latency on every surface | Per-character provider pinning in ModelRegistry (`provider.order`); byte-stability unit tests on every prompt builder in CI; log provider-reported `cached_tokens` on every call; a cache-hit-rate gauge in dev mode; the Week-1 prototype gates everything on ≥80% hits |
| 2 | **WeChat bridge breaks or gets accounts banned** — both puppets are unmaintained; web-protocol logins fail for many post-2017 accounts | High over V1's life / one connector only, by design | Connector-as-plugin isolation with `health()` states surfaced in Config; ship the `GatewayConnector` conformance test suite in the MIT SDK so the community can maintain replacements; experimental labeling (Owner Decision 1); Telegram is the flagship channel |
| 3 | **Event-loop stalls degrade streaming** — synchronous SQLite writes during reflection fan-out, a heavy FTS5 query, or a plugin hot loop can freeze SSE flushes mid-narration | Medium / perceived latency only, never data loss | Keep every transaction under a few milliseconds; cap workers at 4; sharp already runs off-loop on libvips threads; move FTS5/compaction reads to a `worker_threads` read-only connection if needed; ship an event-loop-lag gauge in dev mode that alarms past 200 ms |
| 4 | **The custom-element plugin boundary proves too weak for deep UI plugins** — e.g. a replacement map needs app context beyond the documented connectors | Medium / plugin story (items 5, 6, 13) | The core owns all VN animation; plugins theme via CSS tokens and replace *whole surfaces* behind connectors; the default map renderer is built strictly against the public connector schema (dogfooding) — if it can't be, the *schema* gets extended before any third party depends on it |
| 5 | **Stale edge dependencies** — `web-push` (2 years), the WeChat puppets, and fast-moving Vite majors sit at the edges of an app meant to run unattended for years | Medium / individual features, not core | Pin all versions exactly; batch updates monthly; `web-push` implements a frozen IETF standard (low churn) with `node-webpush` documented as the swap; anything unmaintained lives behind an owned interface (`GatewayConnector`, notifier connector) so replacement never touches core |

## 6. Unified de-risk plan

**Week 1 — the naked hot path, tortured.** Build the smallest real vertical slice: Fastify + SSE + event log + hand-rolled ledger tables (better-sqlite3) + `ContextAssembler` + AI SDK against OpenRouter, running a scripted 3-call scene turn (Narrator → character → narration) streamed sentence-by-sentence to a bare React page — plus a harness that `kill -9`s the process at three injection points (mid-stream, between calls, pre-commit) and restarts it in a loop. No map, no gateway, no plugins. **Success criteria:** (a) first sentence rendered **< 10 s** with a ~50K-token stable prefix on a 256K-class model (Brief §7d); (b) provider-reported `cached_tokens` **≥ 80% of the stable prefix** on turns 2+ across 20 consecutive turns (Brief §2.6); (c) every kill/restart cycle reaches a consistent state with **zero duplicate or lost events**, and a reconnecting client resumes correctly via `Last-Event-ID`; (d) the same stream is consumed by `curl -N` with resume (terminal-client proof, Brief §1); (e) idle RSS **< 150 MB**. **Decision triggers:** (b) fails with pinned providers → revisit item 9 toward direct per-provider clients behind the same ModelRegistry interface; (a) fails on prefill alone → revisit context-size strategy, not the stack; (c) fails → fix the ledger claim/lease SQL before any feature work — items 2, 7, and 8 all stand on it.

**Milestone 2 (weeks 2–3) — the full kill -9 table under load.** Add reflection fan-out at scene end, one painter job (sharp crop → feather → composite under a region lease), grammY long-poll echo with the dedup table, world-clock CRON rows, and a time-skip replay in scheduled-timestamp order. Extend the kill harness to every row of the brief's §4 table: mid-turn, between scene-end and commits, mid-reflection, mid-painter, mid-CRON, client disconnect. **Success criteria:** 100 kill/restart cycles with zero lost/duplicated durable events and zero corrupted images (hash-verified, composite-on-success proven); new-scene opens block only on that world + involved characters' pending jobs; peak RSS **< 256 MB** during fan-out plus one painter composite; time-skip replays code-class instantly and LLM-class in background under the per-skip budget.

**Milestone 3 (weeks 4–6) — plugin proof + packaging.** Build the real VN scene page (sentence pacing, click/Auto-Advance, interrupt-anywhere closing the envelope), the default Canvas 2D map plugin written *only* against public connector events, and then — with the server running — drop a folder into `plugins/` containing one CSS theme, one custom-element surface replacement, and one connector; restart and verify all three load. Produce the Docker image and one Windows archive, and run the update path end-to-end. **Success criteria:** the plugin loads with **zero build step** and its provenance hash appears in dev mode (UI Spec §2.8); the map renderer has no private imports (verified by lint); download-verify-swap-on-restart survives `kill -9` mid-update (idempotent startup swap); scene-open and map-jump animations fully mask the 5–10 s generation window on a mid-range machine and a mobile browser; idle RSS still **< 170 MB** with the three plugins installed.
