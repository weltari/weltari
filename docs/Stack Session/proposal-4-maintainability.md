## B. Coherence check

Everything is one TypeScript codebase on one Node 22 process: Fastify serves the Vite-built React frontend as static files, the `/v1/events` SSE stream, and the TypeBox-validated command API; the custom Scene-Engine, ledger workers, croner scheduler, grammY long-poller, and wechaty bridge are async loops inside that same process, all writing through Drizzle repositories into one WAL SQLite file via the WriteGate. Plugins are ESM on both sides of the same language, so item 13 needs no cross-language bridge; the map renderer is just the first-party instance of that plugin contract; `@weltari/protocol` (TypeBox → JSON Schema) is simultaneously the frontend's types, the plugin SDK's types, and the MIT-licensed artifact external game clients and the V1.5 CLI generate bindings from — no private side-channels by construction. Packaging bundles exactly this: one JS bundle + two native addons + Node runtime. **Idle RAM estimate:** Node 22 baseline ~60–75 MB RSS, Fastify + Drizzle + grammY + engine ~30–40 MB, SQLite page cache ~16 MB, sharp/libvips lazy-loaded (0 idle, ~40–60 MB during painter jobs), wechaty ~15–25 MB when paired → **~125–160 MB idle core**, comfortably under the 256 MB typical target with painter spikes as the transient exception and plugins spending the remaining headroom (§1).

## C. Constraint audit (Brief §2, ten constraints)

1. **Event log as truth:** append-only `event_log` table via `event_log.append()`; SSE ids are log seq numbers; all projections rebuildable from it. Satisfied.
2. **Job Ledger:** item 8 implements states/idempotency/leases/dead-letter/per-world concurrency literally as columns + claim query. Satisfied.
3. **Single writer per entity:** per-character serial queues → single WriteGate connection; `version` columns elsewhere via Drizzle `WHERE version = ?` checks that throw on 0 rows (loud). Satisfied.
4. **Crash-only:** intent rows before work (job claimed = row, turn envelope = row); better-sqlite3 transactions make `scene.ended`+enqueue atomic; startup path = scan `running` jobs with expired leases. Satisfied — the one strain: synchronous writes must stay small (see Risk 3).
5. **Server-pushed stream, render-only frontend:** SSE + zustand stores writable only by the SSE reducer; no fetch-on-mount anywhere; dev channel rides the same stream. Satisfied.
6. **Prompt-prefix ordering:** owned ContextAssembler with byte-stability unit tests + a dev-mode cache-hit metric. Satisfied, but it's discipline-in-code — the prototype (E) verifies it.
7. **Repository layer:** Drizzle confined to `src/repo/**` by lint rule; modules see interfaces only; Postgres swap = new Drizzle dialect behind the same interfaces. Satisfied.
8. **`actor_id` everywhere:** column on `event_log`, `proposals`, `profiles`; TypeBox event envelope makes it a required field — omission fails schema validation loudly. Satisfied.
9. **Proposal pipeline:** one `Proposal` TypeBox schema, one React `<ProposalCard>` diff component, `proposal.created`/`proposal.resolved` events. Satisfied.
10. **Corollaries:** every tool call passes a TypeBox validator before any repository write (LLM output never directly durable); knowledge-tier filter and private channel live in the ContextAssembler; monotonic world clock is an engine-owned table driving croner scheduling and skip replay. Satisfied.

## D. Top 3 risks

1. **WeChat bridge breaks or gets the account banned.** The [wechat4u puppet](https://github.com/wechaty/puppet-wechat4u) is web-protocol-based; post-2017 accounts often can't log in and bans are reported with no known fix — likelihood **high** over V1's life. Blast radius: WeChat channel only — the connector interface, health events, and dedup table isolate it; Telegram and core are untouched (§7c accepted this). Mitigation: `health()` surfaces `degraded/dead` in Config + as an event; the connector-as-plugin format means a replacement bridge (padlocal, community puppet) installs without a core release; ship Telegram as the reference connector so the abstraction is proven by having two implementations day one.
2. **Prompt-cache misses blow the token budget.** If any assembler leaks dynamic content into the stable prefix (an AI-written edit reordering blocks is exactly the failure mode), the 1–5M tokens/day target (§7d) can multiply silently — likelihood **medium**. Blast radius: cost + latency across every surface. Mitigation: byte-stability unit tests on every prompt builder in CI, a cache-hit-rate counter in dev mode, and the prototype's ≥80% hit criterion before any other feature lands (retrofit means rewriting all builders, §2.6 — so verify first).
3. **Event-loop stalls in the single process.** Synchronous better-sqlite3 writes during reflection fan-out bursts, or a plugin's hot loop, freeze SSE streaming for everyone — likelihood **low-medium**, blast radius: perceived latency, not correctness (crash-only protects data). Mitigation: keep every transaction < a few ms (small writes by design), sharp already off-loop, worker pool capped at 4, an event-loop-lag gauge exported to dev mode that *alarms loudly* past 200 ms; escape hatch documented: move painter/reflection workers to `worker_threads` with their own read connections without changing the ledger contract.

## E. De-risk prototype (build first, ≤1 week of AI-agent work)

**A vertical scene-turn slice:** Fastify + SSE + event log + minimal custom engine loop running a real 3-call sequential turn (Narrator → character → narration) through AI SDK v5 against OpenRouter, ContextAssembler with stable-prefix ordering, sentence-split streaming into a bare React page with click-to-advance — plus a `kill -9` harness that murders the process mid-call and restarts it.

**Success criteria (all measurable):** (a) time-to-first-sentence < 10 s (§7d); (b) prompt-cache hit rate ≥ 80% on calls 2–3 across 20 consecutive turns (read from provider usage fields); (c) after `kill -9` at three injection points (mid-stream, between calls, pre-commit), restart reaches a consistent state from the log with zero duplicate events; (d) the same stream is consumed correctly by `curl -N` with `Last-Event-ID` resume (CLI/engine-as-a-service proof, §1).

**Revisit triggers:** (b) fails → item 9: AI SDK's request shaping isn't byte-stable for the target providers; drop to a thin raw-HTTP client behind the same ModelRegistry interface. (c) fails → items 4/7: the envelope/commit design, not the libraries. (a) fails on OpenRouter routing latency → item 9's routing config gains per-function provider pinning, not a stack change. (d) fails → item 4: fix before any frontend work, since the protocol is the product boundary.
