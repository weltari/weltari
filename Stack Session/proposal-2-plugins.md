## B. Coherence check

Everything is one language and one process: TypeScript on Node 24 runs Fastify, the custom Scene-Engine loop, the hand-rolled SQLite ledger (better-sqlite3), sharp, grammY and wechaty in-process; the frontend is Lit-built static files served by the same Fastify instance, and plugins are folders of ES modules that both halves load natively — no FFI, no second runtime, no build step for plugin authors. The protocol package (TypeBox → JSON Schema, MIT) is generated from the same types the server validates with, so the built-in frontend, the V1.5 CLI, and future external game clients all consume one versioned contract over SSE+POST. Packaging (zip-with-runtime / Docker) preserves a real filesystem, which the folder-based plugin format requires. **Estimated idle RAM:** Node 24 baseline ~55 MB + Fastify/app code ~25 MB + better-sqlite3 caches ~15 MB + grammY long-poll ~5 MB + wechaty bridge ~30 MB + sharp/libvips (lazy, idle) ~10 MB ≈ **140–170 MB idle**, inside the 256 MB typical target with headroom for plugins to spend (§1).

## C. Constraint audit (Brief §2, one line each)

1. **Event log as truth:** `event_log` repository is append-only; SSE ids = log seq; all projections rebuild from it — no library owns state (custom loop, no LangGraph checkpointer).
2. **Job Ledger:** hand-rolled table has states, `idempotency_key UNIQUE`, `lease_until`, attempts→`parked`, per-world SQL guard (item 8) — every field of §2.2 is a literal column.
3. **Single writer / mailboxes:** synchronous better-sqlite3 on one event loop + per-character async queues; `version` columns checked in repository `UPDATE … WHERE version=?`.
4. **Crash-only:** migrations + lease-expiry sweep + ledger resume all run in the one startup path; restart-swap updates and composite-on-success painter jobs assume kill -9. *Honest strain:* a sync driver blocks the loop during big transactions — mitigated by keeping transactions small; acceptable single-user.
5. **Server-pushed stream:** SSE carries narration, dev channel, gateway echoes, backpack, proposals on every page; frontend components are render-only custom elements; no polling anywhere.
6. **Prompt-prefix ordering:** one `ContextAssembler` class is the only prompt builder; AI SDK receives already-ordered messages; unit test asserts byte-stable prefixes across turns.
7. **Repository layer:** ESLint `no-restricted-imports` makes repositories the only SQL-touching modules — the Postgres swap is a driver rewrite behind unchanged interfaces.
8. **`actor_id`:** present in the events table schema, every TypeBox event/command type, and the jobs table — schema-enforced, not convention.
9. **Proposal pipeline:** one `Proposal` TypeBox type, one `<weltari-proposal>` Lit component, approve/reject POST commands logged as events.
10. **Corollaries:** engine validates every AI-SDK-parsed tool call before any repository write; knowledge-tier filtering lives in the assembler; world clock is an engine-owned monotonic column driving `scheduled_game_ts`.

## D. Top 3 risks

1. **WeChat bridge breaks or bans accounts** (wechat4u web-API limits, documented ban reports — [puppet-wechat4u](https://github.com/wechaty/puppet-wechat4u), [wechaty docs](https://wechaty.js.org/docs/puppet-providers/wechat4u)). *Likelihood:* high over a year. *Blast radius:* WeChat channel only — the connector-as-plugin boundary (items 11/13) means Telegram and the core never notice. *Mitigation:* ship the `GatewayConnector` conformance test suite in the MIT SDK so the community can maintain replacement bridges; document the ban risk in onboarding; keep Telegram the flagship channel.
2. **Custom-element plugin frontend proves too weak for the VN surface** (deep animation coordination across plugin/core boundary). *Likelihood:* medium. *Blast radius:* item 5, and item 13's frontend half. *Mitigation:* the core owns all VN animation; plugins theme via CSS tokens and replace *whole surfaces* (map, feed) behind connectors rather than injecting into mid-animation trees — validated in the prototype (E).
3. **Event-loop stalls under concurrent load** (sync SQLite writes + sharp compositing while streaming narration) breach the 5–10 s first-content budget. *Likelihood:* medium-low (LLM latency dominates). *Blast radius:* hot-path feel only. *Mitigation:* sharp already runs on libuv threads off-loop; cap ledger workers at 4; if stalls appear, move painter compositing to one `worker_thread` — a contained change to item 10.

## E. De-risk prototype

**Build the "one scene turn + one drop-in plugin" vertical slice:** Fastify + SSE stream + custom loop running a real 3-call turn (Narrator → character → narration) through the ContextAssembler against OpenRouter, persisted to event log + ledger via better-sqlite3, rendered by a minimal Lit scene page — **then, with the server still running, drop a folder into `plugins/` containing one CSS reskin + one custom-element replacement for the narration panel, restart, and see both applied.** Success criteria: first sentence streamed **< 10 s** (§7d); provider-reported prompt-cache hit rate **≥ 80 %** on turns 2–5; `kill -9` mid-turn then restart replays cleanly from `Last-Event-ID`; the plugin loads with **zero build step** and its provenance hash shows in a stub config view. If cache hits stay < 50 %, revisit item 9 (assembler/provider choice); if the runtime custom element can't intercept the narration surface cleanly, revisit item 5 before any other frontend work exists.
